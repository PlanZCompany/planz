# PlanZ – Launch Checklist

Този файл описва стъпките преди и след пускането на сайта, за да не забравяме важни неща (SEO, инфраструктура, процес).

---

## 1. Финален преглед на сайта

### 1.1. Текстове и съдържание

- [ ] Преглед на всички секции на BG:
  - [ ] Hero
  - [ ] Услуги (Services)
  - [ ] Процес (Process)
  - [ ] Примери / Портфолио
  - [ ] Резултати (Results)
  - [ ] Tech & начин на работа
  - [ ] FAQ
  - [ ] Контакт
- [ ] Преглед на EN версията за същите секции
- [ ] Проверка за:
  - [ ] правопис/граматика,
  - [ ] консистентност на термини (напр. „клиент“, „бизнес“),
  - [ ] реални услуги и реални срокове (да не обещаваме повече, отколкото можем).

### 1.2. Изображения и mockups

- [ ] Всички изображения са с разумен размер (Web, не оригиналните огромни).
- [ ] Alt текстовете:
  - [ ] описват какво има на изображението,
  - [ ] не са keyword spam.
- [ ] Device mockups за проекти:
  - [ ] показват реални екрани от наша работа,
  - [ ] не изглеждат прекалено „stock“.

### 1.3. UI/UX basic check

- [ ] Навигацията работи (desktop + mobile).
- [ ] Hero CTA скролва/води към логична следваща секция или контакт.
- [ ] Няма счупени линкове.
- [ ] Бутоните/линковете имат ясни етикети (не „Click here“).

---

## 2. Инфраструктура и хостинг

### 2.1. Домейн и DNS

- [ ] Домейнът е насочен към правилния хостинг (SuperHosting / друг).
- [ ] `www.` → redirect към naked домейн (или обратното, но избран е един canonical).
- [ ] `http` → redirect към `https`.

### 2.2. SSL / HTTPS

- [ ] SSL сертификатът е активен (Let’s Encrypt или друг).
- [ ] Проверка:
  - [ ] Отваряне на сайта през `https://` без предупреждения.
  - [ ] Всички ресурси (CSS/JS/images) се зареждат по HTTPS (без mixed content).

### 2.3. Build & deploy

- [ ] Успешен build локално (`pnpm build`).
- [ ] Ако static export:
  - [ ] `pnpm export` генерира `out/` успешно.
  - [ ] Съдържанието на `out/` е качено на сървъра.
- [ ] Проверка на production URL-а:
  - [ ] home BG и EN,
  - [ ] ключови секции,
  - [ ] формата за контакт.

---

## 3. Формата за контакт и имейл

- [ ] Формата праща заявки успешно (без грешка).
- [ ] Nodemailer / backend:
  - [ ] правилен SMTP (Gmail / хостинг / друг),
  - [ ] тестово запитване успешно пристига.
- [ ] Имейл шаблонът:
  - [ ] съдържа дата и час,
  - [ ] включва име, имейл, бизнес, телефон, сайт, бюджет, съобщение,
  - [ ] няма чувствителна информация в логове.
- [ ] Благодарственото съобщение (success state) се показва:
  - [ ] ясно съобщение към потребителя,
  - [ ] без да се разчита само на alert().

---

## 4. SEO – базови технически неща

### 4.1. Meta данни

- [ ] `title` и `description` са зададени за:
  - [ ] `/` (BG)
  - [ ] `/en` (EN)
- [ ] Open Graph:
  - [ ] `openGraph.title` и `openGraph.description` са смислени,
  - [ ] `openGraph.url` сочи към правилния URL,
  - [ ] `openGraph.images` използва коректни пътища (`/static/...`).
- [ ] Twitter:
  - [ ] `card: 'summary_large_image'`,
  - [ ] коректни `title`, `description`, `images`.

### 4.2. OG изображения

- [ ] OG изображенията са в правилен аспект (`1200x630` или еквивалент).
- [ ] Текстът на тях:
  - [ ] не е прекалено дребен,
  - [ ] четим на мобилно/desktop preview.
- [ ] Teст:
  - [ ] Facebook Sharing Debugger,
  - [ ] Twitter Card Validator,
  - [ ] Discord/Slack линк превю.

### 4.3. Robots + Sitemap

- [ ] `public/robots.txt`:
  - [ ] `Allow: /`,
  - [ ] `Sitemap: https://real-domain.com/sitemap.xml`.
- [ ] `public/sitemap.xml`:
  - [ ] съдържа всички важни URL-и (`/`, `/en` и др.),
  - [ ] URLs са с HTTPS и реалния домейн.

### 4.4. JSON-LD (структурирани данни)

- [ ] `Organization` JSON-LD:
  - [ ] име на организацията,
  - [ ] URL,
  - [ ] имейл за контакт,
  - [ ] по желание – лого.
- [ ] `WebSite` JSON-LD:
  - [ ] име,
  - [ ] URL,
  - [ ] `inLanguage`,
  - [ ] `potentialAction` по желание (search).

---

## 5. Analytics и Search Console

### 5.1. Analytics

- [ ] Избран инструмент:
  - [ ] Google Analytics 4 **или** друга платформа (Plausible, Fathom и т.н.).
- [ ] Скриптът е:
  - [ ] зареден само в production,
  - [ ] асинхронен и не блокира рендера.
- [ ] Проверка:
  - [ ] посещение от инкогнито,
  - [ ] вижда се в real-time dashboard (ако ползва GA4).

### 5.2. Google Search Console

- [ ] Добавен домейн (Domain property или URL prefix).
- [ ] Ownership потвърден (DNS TXT, HTML файл или meta tag).
- [ ] Подаден sitemap (`/sitemap.xml`).
- [ ] Проверка за:
  - [ ] грешки в Coverage,
  - [ ] проблеми с hreflang (ако има).

---

## 6. UX и достъпност (quick pass)

- [ ] Проверка на сайта на мобилно устройство:
  - [ ] навигацията е удобна,
  - [ ] текстовете са четими,
  - [ ] бутоните са достатъчно големи.
- [ ] Контраст на основните текстове/бутони е ок.
- [ ] Основните линкове и бутони могат да се ползват и с клавиатура (Tab).

---

## 7. Маркетинг и присъствие

### 7.1. Базови линкове към сайта

- [ ] Линк в:
  - [ ] LinkedIn (лични профили),
  - [ ] LinkedIn company page (ако има),
  - [ ] GitHub профили,
  - [ ] други релевантни профили (Behance, Dribbble и т.н.).
- [ ] Подпис в имейл:
  - [ ] име,
  - [ ] роля,
  - [ ] линк към PlanZ сайта.

### 7.2. Първоначален outreach

- [ ] Списък с 3–5 познати бизнеса/контакти, на които да изпратите:
  - [ ] линк към сайта,
  - [ ] кратко представяне какво предлагате,
  - [ ] предложение за кратък call.

---

## 8. Продажбен процес (internal)

### 8.1. След запитване от формата

- [ ] Време за реакция: цел ≤ 24 часа.
- [ ] Имейл шаблон за първи отговор:
  - [ ] благодарност,
  - [ ] потвърждение, че сте получили запитването,
  - [ ] предложение за 2–3 слота за кратък call.
- [ ] Списък с въпроси за първия call:
  - [ ] тип бизнес,
  - [ ] основна цел на сайта,
  - [ ] примерни конкуренти/референции,
  - [ ] бюджетен диапазон,
  - [ ] срокове/очаквания.

### 8.2. Оферти и обхват

- [ ] Базов темплейт за оферта (Google Docs / Notion / PDF):
  - [ ] резюме на проблема,
  - [ ] какво предлагаме (обхват),
  - [ ] срокове,
  - [ ] цена/диапазон,
  - [ ] какво НЕ включва.
- [ ] Ясно вътрешно правило:
  - [ ] кога казваме „да“,
  - [ ] кога отказваме (твърде малък бюджет / несъвместим обхват / нереалистични срокове).

---

## 9. След launch – 1–3 месеца по-късно

- [ ] Проверка в Google Search Console:
  - [ ] кои ключови думи водят трафик,
  - [ ] CTR по страници (дали има нужда от пипане на title/description).
- [ ] Проверка в Analytics:
  - [ ] кои секции държат хората най-дълго,
  - [ ] къде има голям drop-off.
- [ ] Обновяване на портфолиото:
  - [ ] нови 1–2 проекта,
  - [ ] по 1–2 изречения за резултатите.
- [ ] Събиране на feedback от реални клиенти:
  - [ ] 2–3 кратки testimonials,
  - [ ] евентуално добавяне на секция „Отзиви“ на сайта.

---
